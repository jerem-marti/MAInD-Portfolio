# Internationalization (i18n)

> How the English-default / French-under-`/fr` system is wired: where every translatable string lives, how to add a French page or string, and how to prove `/fr` has no English leaks.

## At a glance

| I need to… | Edit | Verify |
|---|---|---|
| Translate / add a UI string | `i18n/locales/en.json` (add key — source of truth) **then** `i18n/locales/fr.json` (same key path) | `npm run generate && npm run check:i18n` |
| Add a French twin for a case study | `content/fr/work/<slug>.md` (mirror of `content/en/work/<slug>.md`) | open `.output/public/fr/projets/<slug>/index.html`; `npm run check:i18n` |
| Translate a study's home-page card (problem/outcome/title/alt) | `i18n/locales/{en,fr}.json` under `data.featured.<slug>` or `data.projects.<slug>` | `npm run check:i18n` |
| Change a French URL segment | `nuxt.config.ts` `i18n.pages` map (`nuxt.config.ts:58`) + `modules/llms.ts` route lists + `scripts/check-i18n-leaks.mjs` pairs | `npm run generate && npm run check:i18n` |
| Translate an llms.txt / `.md` artifact string | `i18n/locales/fr.json` under `llms.*` (English defaults are inline in `server/llms/builders.ts`) | `npm run generate`; open `.output/public/fr/llms.txt` |
| Diagnose "string still English on `/fr`" | see [Runbook: a string is still English](#runbook-a-string-is-still-english-on-fr) | `npm run check:i18n` |

## How it works

The site is static (`nitro.preset: 'static'`, `nuxt.config.ts:27`) and bilingual via `@nuxtjs/i18n` v10. English is the default at `/` (unprefixed); French lives under `/fr` with French URL segments.

| Concern | Setting | Location |
|---|---|---|
| Module order | i18n loads **before** `@nuxtjs/seo` so sitemap/og/hreflang see the locales | `nuxt.config.ts:11` |
| Strategy | `prefix_except_default` — EN at `/`, FR at `/fr` | `nuxt.config.ts:56` |
| Locales | `en` (`language: 'en'`, `en.json`) and `fr` (`language: 'fr'`, `fr.json`); `language` keeps `<html lang>` as plain `en`/`fr` | `nuxt.config.ts:62-65` |
| Custom routes | `customRoutes: 'config'` → translated paths come from the `pages` map, not from components | `nuxt.config.ts:57-61` |
| URL remaps | `about → /a-propos`, `work/[slug] → /projets/[slug]` (`contact` and `index` are **not** remapped) | `nuxt.config.ts:58-61` |
| Lazy catalogs | `lazy: true` + `experimental.prerenderMessages: true` ships the JSON catalogs as static assets (no Nitro route on a static host) | `nuxt.config.ts:66,70` |
| Browser redirect | `detectBrowserLanguage: false` — Nuxt does **no** language redirect; nginx owns the root `/` redirect | `nuxt.config.ts:67` |
| SEO | `experimental.strictSeo: true` — i18n owns hreflang / `x-default` / canonical / `og:locale`, prerender-safe | `nuxt.config.ts:69` |
| Runtime behavior | `legacy: false` (Composition API), `fallbackLocale: 'en'` (missing FR key → English, never a raw key) | `i18n/i18n.config.ts:5-8` |

Resulting URL map:

```
EN (default)        FR (prefixed)
/                   /fr
/about              /fr/a-propos
/contact            /fr/contact
/work/<slug>        /fr/projets/<slug>
```

**The root redirect is nginx, not Nuxt.** A first-time visitor whose `Accept-Language` leads with French and who lands on `/` is sent to `/fr` once (302), with a sticky cookie so switching back to EN is not re-redirected. Every other URL is served verbatim, so bots and deep links are never redirected. The rule is the `map` + `location = /` block in `deploy/nginx-i18n.conf:16-66`.

**Sitemap.** `@nuxtjs/seo` emits a per-locale sitemap index at `/sitemap_index.xml → /__sitemap__/{en,fr}.xml`. The localized `/fr/sitemap.xml` redirect stub is junk on a static host, so it is dropped via `nitro.prerender.ignore` (`nuxt.config.ts:34`). llms artifacts are kept out of the sitemap for both locales (`nuxt.config.ts:168-171`).

## The message catalog

Every user-facing string flows through the catalog. There are two files:

- `i18n/locales/en.json` — **the single source of truth.** Top-level groups: `skipToContent`, `nav`, `chrome`, `footer`, `common`, `home`, `about`, `contact`, `error`, `work`, `tags`, `data`, `og`, `schema`.
- `i18n/locales/fr.json` — French translations under the **same key paths**, plus one FR-only group `llms` (see below). Any key absent from `fr.json` falls back to the English string per `fallbackLocale: 'en'` (`i18n/i18n.config.ts:7`).

Notable groups:

- `tags.*` — the Index controlled vocabulary, keyed by the English label (identity-mapped in `en.json`).
- `data.featured.<slug>` / `data.projects.<slug>` — the **display strings** for home-page cards (`title`, `problem`, `outcome`, `meta`, `alt`). These live in the catalog, **not** in `app/data/*.ts` (those files keep only structure: id / slug / image / tags / year).
- `og.*`, `schema.*` — share-card and schema.org strings.
- `llms.*` (**`fr.json` only**) — French overrides for the llms.txt / `.md` artifacts. The English literals are inline defaults in `server/llms/builders.ts`; `fr.json`'s `llms.*` overrides them. There is intentionally **no `llms` key in `en.json`**.

How components consume strings:

- Plain text: `const { t } = useI18n()` then `t('home.selectedWorkLabel')` (e.g. `app/pages/work/[slug].vue:7,168`).
- A paragraph with one highlighted phrase: `<i18n-t>` with an `{emphasis}` slot. The sentence lives in one key (e.g. `about.heroBody`), the highlighted run in a sibling key (`about.heroEmphasis`), so word order can differ between locales:

  ```vue
  <i18n-t keypath="about.heroBody" tag="p" scope="global">
    <template #emphasis>
      <span class="…accent">{{ t('about.heroEmphasis') }}</span>
    </template>
  </i18n-t>
  ```
  See `app/pages/about.vue:50-59`. The `{emphasis}` placeholder must appear in **both** `en.json` and `fr.json` for that key.

- The language toggle (footer) uses `useSwitchLocalePath()` to link each locale to the **current page in that locale**, and reads `locales` for the labels (`app/components/chrome/Footer.vue:2-3,43-64`). In-app links use `useLocalePath()` so `localePath('/')` resolves to `/` or `/fr` per active locale (`app/components/chrome/TopNav.vue:3`, `app/pages/work/[slug].vue:8,148`).

## Per-locale case studies

Case studies are two Nuxt Content collections, one per locale (`content.config.ts:123-135`):

- `work_en` — `content/en/work/*.md`, `prefix: '/work'`
- `work_fr` — `content/fr/work/*.md`, `prefix: '/work'`

Both strip the `en`/`fr` segment, so each resolves to the **same internal path** `/work/<slug>`. The visible `/fr/projets/<slug>` URL prefix comes from i18n routing, not from Content.

The page resolves the active locale with a French-then-English fallback, so `/fr/projets/<slug>` always renders even before a French twin exists (`app/pages/work/[slug].vue:15-25`):

```ts
const localized = locale.value === 'fr'
  ? await queryCollection('work_fr').path(`/work/${slug}`).first()
  : null
return localized ?? (await queryCollection('work_en').path(`/work/${slug}`).first())
```

The "More work" prev/next cards apply the same per-study fallback: French `card` where present, English otherwise (`app/pages/work/[slug].vue:35-45`).

The `/fr/projets/<slug>` routes (and their `.md` twins) **auto-enumerate** at build: `modules/llms.ts` globs the live slugs from `content/en/work/*.md` (`modules/llms.ts:25-30`) and pushes `/fr/projets/<slug>` page routes and `/fr/projets/<slug>.md` artifact routes into the prerender set (`modules/llms.ts:44-51,62-67`). Adding an EN study therefore adds its FR route with no module change.

## Runbooks

### How to add a French twin for a study

Precondition: the English study exists at `content/en/work/<slug>.md` and is registered per `docs/content-authoring.md`.

1. Copy `content/en/work/<slug>.md` to `content/fr/work/<slug>.md`.
2. Keep **structure, keys, and all paths identical** (image `src`, `hero`, slugs). Translate only prose: `title` (if it translates), `summary`, `problem[]`, `approach[].{label,title,prose,artifacts[].{alt,caption,decision}}`, `outcome[]`, `reflection`, `gallery[].{alt,caption}`, `role.*`, `brief.*`, `resources[].title`, `card.{title,alt}`. Keep proper nouns, product names, and any deliberate verbatim-French (see [Constraints](#constraints) and `docs/wip/FR-REVIEW.md`).
3. The `/fr/projets/<slug>` route already exists (EN fallback); the twin replaces the fallback with French. No `nuxt.config.ts` or `modules/llms.ts` change is needed.
4. If this is a **new** study (not just a translation), also add its home-page card display strings under `data.featured.<slug>` or `data.projects.<slug>` in **both** `en.json` and `fr.json` — see `docs/content-authoring.md`.
5. Verify:
   ```bash
   npm run generate && npm run check:i18n
   ```
   Confirm `.output/public/fr/projets/<slug>/index.html` exists and reads French.

### How to add / translate a UI string

1. Add the key to `i18n/locales/en.json` under the correct group (source of truth).
2. Add the French value to `i18n/locales/fr.json` at the **same key path**. If you omit it, `fallbackLocale: 'en'` renders the English string on `/fr` — a silent leak.
3. Consume it: `t('group.key')`, or `<i18n-t keypath="group.key" tag="…" scope="global">` with a matching slot for an emphasized run (mirror the placeholder in both catalogs).
4. Do **not** hardcode the string in `app/data/*.ts` — structure only.
5. If the string is unambiguous chrome/UI (always fully translated, never a proper noun), add it to `SENTINELS` in `scripts/check-i18n-leaks.mjs:20-37` so a future English leak on `/fr` is caught automatically.
6. Verify: `npm run generate && npm run check:i18n`.

### How to add a French URL segment

1. Add/edit the entry in the `i18n.pages` map (`nuxt.config.ts:58-61`). The key is the **file-based route name** (`about`, `work/[slug]`); the value is `{ fr: '/<french-segment>' }`.
2. If it introduces a **new prerendered tree** (a new dynamic route or artifact), mirror the paths in `modules/llms.ts` (`frPageRoutes` `modules/llms.ts:62-67` and, for `.md` twins, `frArtifactRoutes` `modules/llms.ts:44-51`) so the whole `/fr` tree prerenders even if a link is missed.
3. Update the route-parity pairs in `scripts/check-i18n-leaks.mjs:77-82` and the per-locale `sitemap.exclude` artifact paths in `nuxt.config.ts:168-171` if new artifact paths appear.
4. Verify: `npm run generate && npm run check:i18n`.

### Runbook: a string is still English on /fr

Check in this order:

1. **Missing FR key** — the key exists in `en.json` but not `fr.json`; `fallbackLocale: 'en'` shows English. Add it to `fr.json`.
2. **Hardcoded string** — the component renders a literal instead of `t(...)`, or a display string was put in `app/data/*.ts`. Move it into the catalog.
3. **Deliberate verbatim-keep** — some French copy keeps English on purpose (e.g. `chrome.tagline` is `"Interaction and product designer"` in `fr.json`; the home/about ledes open with the same English job title). Confirm against `docs/wip/FR-REVIEW.md` before "fixing."
4. **Missing FR content twin** — a case-study page shows English because `content/fr/work/<slug>.md` doesn't exist (EN fallback). Add the twin.
5. Run `npm run check:i18n` — it names every English-sentinel leak with its file path.

## Constraints

- **`en.json` is the only source of truth for display strings.** Add the English key first, then the French.
- **Never hardcode display strings in `app/data/*.ts`.** Those files hold structure (id / slug / image / tags / year); prose lives under `data.*` / `tags.*` in the catalog.
- **Keep verbatim French and proper nouns.** Product/place/school names (DataBloom, WeMatch, Thea, SUPSI, HEIG-VD, EPFL, HEIA-FR, Baleinev) and deliberate English phrases stay as-is. The owner-reviewed glossary, voice rules, and verbatim-keep list are in `docs/wip/FR-REVIEW.md`; idiomatic-rewrite notes are in `docs/wip/FR-DECALQUAGE.md`.
- **Swiss formatting / voice carries into French**: thousands separator `'` (e.g. `1'000`), US-style restraint, no em-dashes in body prose (see `docs/brand-voice.md`).
- **The site is static.** No runtime locale negotiation in Nuxt; the only language redirect is the nginx root rule (`deploy/nginx-i18n.conf`).
- **French is owner-reviewed.** Treat existing FR as a calque to replace with idiomatic French, not a baseline to protect (per `docs/wip/FR-DECALQUAGE.md`).

## Verify

```bash
npm run generate && npm run check:i18n
```

`check:i18n` runs `scripts/check-i18n-leaks.mjs` against `.output/public` and exits non-zero on any failure. It asserts three things:

1. **No English leak on `/fr`** — every `/fr/**.html` page's visible text is scanned for ~16 fully-translated UI sentinels (`scripts/check-i18n-leaks.mjs:20-37`); any hit fails.
2. **Route parity** — every EN route has its FR counterpart at the custom segment: `index ↔ fr/index`, `about ↔ fr/a-propos`, `contact ↔ fr/contact`, and each `work/<slug> ↔ fr/projets/<slug>` (`scripts/check-i18n-leaks.mjs:74-86`).
3. **`<html lang>` + canonical** — each prerendered page carries the correct `<html lang="en|fr">` and a `<link rel="canonical">` (`scripts/check-i18n-leaks.mjs:89-102`).

> **Local environment:** on this machine, npm commands may fail TLS verification (`UNABLE_TO_VERIFY_LEAF_SIGNATURE`). Prefix with `NODE_OPTIONS=--use-system-ca` if needed (e.g. `NODE_OPTIONS=--use-system-ca npm run generate`). CI/other machines do not need this.
